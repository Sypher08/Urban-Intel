01Test Suite Overview

Playwright E2E tests for the Urban Intel emergency response platform — covering every user role from login through dispatch to incident resolution.

Architecture

  • Driver: Playwright 1.61.0, Chromium only
  • Target: Expo web build at http://localhost:8081
  • Backend: FastAPI / uvicorn at http://localhost:8001
  • Parallelism: fullyParallel: true — each test gets its own browser context
  • Auto-start: webServer config boots both servers via Playwright

export const DEMO_CREDENTIALS = {
  citizen: { email: "citizen@urbanintel.app", password: "Citizen@123" },
  admin:   { email: "admin@urbanintel.app",   password: "Admin@123" },
  fire:    { email: "fire@urbanintel.app",    password: "Fire@123" },
  medical: { email: "medical@urbanintel.app", password: "Medical@123" },
  police:  { email: "police@urbanintel.app",  password: "Police@123" },
};

export async function login(page, email, password) {
  await page.goto("/auth/login", { waitUntil: "networkidle" });
  await page.waitForSelector('[data-testid="login-email"]');
  await page.fill('[data-testid="login-email"]', email);
  await page.fill('[data-testid="login-password"]', password);
  await page.click('[data-testid="login-submit"]');
}

export async function clearSession(page) {
  await page.goto("/auth/login", { waitUntil: "domcontentloaded" });
  // Must be on same-origin before clearing localStorage
  await page.evaluate(() => { localStorage.clear(); sessionStorage.clear(); });
  await page.context().clearCookies();
}
clearSession() navigates to /auth/login before touching localStorage. This avoids the SecurityError: localStorage is not available for opaque origins that occurs on about:blank.

09Selector Strategy: testID → data-testid

The single most important debugging finding — why all 38 tests fail and how it was fixed.

Root cause: All tests use [testID="..."] CSS selectors, but React Native Web renders the testID prop as data-testid in the DOM. The selectors never match any element, causing every assertion to fail with locator('[testID="..."]') not found.

How React Native Web Handles testID

Located in react-native-web/dist/modules/createDOMProps/index.js:

// line 151 — destructured from props
testID = _props.testID;

// line 835-836 — written as data-testid in DOM
if (testID != null) {
  domProps['data-testid'] = testID;
}

This means following JSX:

<TextInput testID="login-email" />

Renders to HTML:

<input data-testid="login-email" ... />

NOT:

<input testID="login-email" ... />  // ✗ never happens

Fix Applied

All 5 spec files + helpers.ts were updated using a bulk replacement:

# Before
page.waitForSelector('[testID="login-email"]');
page.locator('[testID="login-email"]');
page.locator('[testID^="inc-"]');

# After
page.waitForSelector('[data-testid="login-email"]');
page.locator('[data-testid="login-email"]');
page.locator('[data-testid^="inc-"]');
Playwright's built-in page.getByTestId("login-email") also works since it matches data-testid by default. The CSS [data-testid="..."] approach was chosen for consistency with existing code style.

10Known Issues & Fixes

Every failure encountered during test development, with resolution.

Issue Error Root Cause Fix
testID selector mismatch locator('[testID="..."]') not found React Native Web renders testID as data-testid Replace all [testID= → [data-testid=
clearSession SecurityError SecurityError: localStorage is not available for opaque origins page.goto("about:blank") creates a cross-origin context where localStorage is inaccessible Navigate to /auth/login first (same origin) before calling localStorage.clear()
waitForURL regex mismatch Timeout: page.waitForURL Regex patterns like /\/(citizen)\/home/ don't match when Expo Router collapses route groups in URL Use simpler /\/home$/, /\/dashboard$/ patterns
text=Create account strict mode strict mode violation: resolved to 2 elements Both the page heading and the submit button contain "Create account" text Use text=Create account >> visible=true or a more specific selector (pending)
Server not reachable net::ERR_CONNECTION_REFUSED webServer command path is relative and may fail on Windows if not run from correct cwd Verify cwd paths in config; use reuseExistingServer: true with pre-started servers
page.goto wrong path 404 / wrong page incident-lifecycle.spec.ts used /report instead of /home for citizen home Fixed paths to match Expo Router file conventions

Permanent countermeasures

DONE clearSession

Always navigates to same-origin page before touching storage. This is now enforced by the shared helper.

DONE Selector strategy

All future e2e selectors must use [data-testid="..."] or page.getByTestId(), never [testID="..."].

TODO "Create account" selector

Needs disambiguation — likely h2:has-text("Create account") for heading and button:has-text("Create account") for button.

TODO webServer on Windows

May need absolute cwd paths or fall back to start-servers.ps1 for manual boot on Windows.

11How to Run the Tests

Two methods — manual server start or Playwright auto-start.

Method 1 — Auto-start (recommended)

# From project root — Playwright boots both servers automatically
npx playwright test --reporter=list

# Run a single spec file
npx playwright test e2e/auth.spec.ts --reporter=list

# Run with HTML report (auto-opens)
npx playwright test
npx playwright show-report

Method 2 — Manual start

Start backend

In Terminal 1: cd backend; .venv\Scripts\Activate.ps1; uvicorn server:app --port 8001 --reload

Start frontend

In Terminal 2: cd frontend; yarn web --port 8081

Run tests

Wait for both to be ready, then in Terminal 3: npx playwright test

Using the convenience script

# Start both servers
.\e2e\start-servers.ps1 -Start

# In another terminal, run tests
npx playwright test

# Stop both servers
.\e2e\start-servers.ps1 -Stop
