01Project Overview

A complete, production-style emergency response platform — not a toy demo.

Urban Intel lets any citizen submit an emergency report from a mobile / web app with a single photo and one tap. A vision LLM (Gemini) automatically classifies the incident type, estimates severity, and recommends which combination of services to dispatch — Ambulance, Fire, Police, or all three. Reports land in a Supabase Postgres database; agency operators see them in real-time on a Dispatch Console that auto-refreshes every 10 seconds; citizens can stream their live GPS position to dispatch until help arrives.

Why this is industry-level (not just MVP)

  • AI triage + user input fusion — final severity = 40 % user + 60 % AI (no single point of failure).
  • Multi-service dispatch — building fire ⇒ Fire + Ambulance + Police simultaneously.
  • Live GPS streaming — citizen pings every 12 s while moving (e.g. escaping fire); dispatch sees the trail.
  • Mismatch warning — if user picks "Police" but AI sees a fire, dispatch is alerted.
  • Role-based access control — citizen / agency-by-type / admin enforced at FastAPI dependency layer.
  • Graceful AI fallback — if Gemini is down, app still submits and routes by user input.
  • Idempotent seed users — restarting the backend never duplicates data.

04Role-Specific Workflows

What each user sees and does — citizen, fire, ambulance, police, admin.

📱

Citizen

Reports the emergency. Streams live location. Watches status update.

What the citizen actually does

  • Opens the app → if previously signed in, JWT in expo-secure-store auto-resumes the session.
  • Sees the SOS pulse on the home screen. One tap = capture GPS + auto-open camera + pre-fill severity=High.
  • For non-SOS reports: picks photo, taps "Run AI Triage" — sees Gemini's incident_type, confidence %, recommended_services, plus a yellow mismatch banner if their pick disagrees with the AI.
  • Submits — gets routed to the incident detail page. Can tap "Share live location with dispatch" to begin GPS streaming.
  • Detail screen polls every 15 s and shows the response timeline advance in real time.
🔥

Fire Dispatch

Receives fire-tagged incidents. Acknowledges, assigns truck, transitions status until resolved.

Fire-only filter

When the agency operator logs in with agency_type=Fire, the server-side query on GET /api/incidents automatically adds: 

sb.table("incidents")
   .contains("recommended_services", ["Fire"])
   .order("created_at", desc=True)

So the Fire console only sees incidents the AI tagged as needing a fire truck — building fires, smoke, vehicle fires, wildfires.

🚑

Ambulance Dispatch

Handles medical incidents — injuries, accidents (with casualties), cardiac, fire-with-injured.

Same 5-stage flow as Fire (New → Acknowledged → En Route → On Scene → Resolved), but filtered server-side to recommended_services ∋ Ambulance. Vehicle codes get auto-generated as UI-#### on acknowledge. ETA is randomized 4-15 min for the demo; in production it would come from a routing service (OSRM / Google Directions).

👮

Police Dispatch

Handles crime, weapons-visible reports, road accidents (always co-dispatched).

Identical workflow. Filter: recommended_services ∋ Police. When the AI sees a road accident it returns ["Ambulance","Police"], so a single incident lands on both Police and Ambulance consoles simultaneously — true multi-service dispatch.

📊

City Admin

Sees everything. Audits database. Runs analytics. Opens this dossier.

Admin role bypasses agency-type filters and sees all incidents. The admin's "server icon" in the dashboard header opens /db-admin, which calls GET /api/admin/db-overview — a live snapshot of every Supabase collection with the 20 most-recent rows (password hashes always scrubbed). From there a banner opens this HTML dossier.

06Database — Supabase / Postgres

Two tables. UUID primary keys. JSONB for AI output and live GPS trail.

All writes/reads from the FastAPI backend use the service-role key, which bypasses Row-Level Security. Access control is enforced inside FastAPI via JWT + role dependencies — never trusted to the client.

09AI Triage Specification

How the Gemini classification works, end-to-end.

Model selection (the backend tries these in order)

  1. Google Gemini direct API — when GEMINI_API_KEY is set. Uses google-genai SDK with model gemini-2.5-flash and response_mime_type=application/json for guaranteed JSON.
  2. Emergent gateway — when EMERGENT_LLM_KEY is set. Uses emergentintegrations with gemini-3-flash-preview.
  3. Rule-based fallback — when neither is set or both fail. Returns incident_type=Other, ai_severity=Medium, recommended_services=[user's pick]. The app never blocks.

System prompt

You are an emergency-incident triage AI for a city emergency response platform.
Given a photo of an incident, return STRICT JSON (no markdown):
{
  "incident_type": "Fire | Accident | Medical | Crime | Flood | Disaster | Other",
  "confidence": 0.0-1.0,
  "ai_severity": "Low | Medium | High",
  "recommended_services": ["Ambulance" | "Fire" | "Police"],
  "reasoning": "one short sentence"
}
Rules:
- Fire / smoke  → include "Fire"
- Injuries / medical → include "Ambulance"
- Crime / weapons → include "Police"
- Road accident → ["Ambulance","Police"]
- Building fire → ["Fire","Ambulance","Police"]

Output safety pipeline

  1. Strip the data:image/...;base64, prefix.
  2. Parse the first JSON object via regex (r'\{.*\}') → json.loads.
  3. Enum-validate incident_type, ai_severity; coerce recommended_services to allowed values; ensure it's non-empty.
  4. Compute mismatch_warning = "AI recommends X but user selected Y" when user's pick is not in the AI's recommendations.
  5. Any exception → graceful fallback (no 500 to the citizen).

Severity fusion formula

score          = 0.4 × user_severity + 0.6 × ai_severity
                 (Low=1, Medium=2, High=3)
final_severity = round(score) clamped to [1,3] → {Low | Medium | High}

from tensorflow.keras.layers import GlobalAveragePooling2D, Dense, Dropout
from tensorflow.keras.models import Model
from sklearn.metrics import classification_report, confusion_matrix
import numpy as np

IMG, BATCH, EPOCHS = (224, 224), 32, 12
train = tf.keras.utils.image_dataset_from_directory("data/train", image_size=IMG, batch_size=BATCH, label_mode="categorical")
val   = tf.keras.utils.image_dataset_from_directory("data/val",   image_size=IMG, batch_size=BATCH, label_mode="categorical")
classes = train.class_names

base = MobileNetV2(weights="imagenet", include_top=False, input_shape=IMG + (3,))
base.trainable = False
x = GlobalAveragePooling2D()(base.output)
x = Dropout(0.3)(x)
x = Dense(128, activation="relu")(x)
out = Dense(len(classes), activation="softmax")(x)

model = Model(base.input, out)
model.compile(optimizer="adam", loss="categorical_crossentropy",
              metrics=["accuracy", tf.keras.metrics.Precision(), tf.keras.metrics.Recall()])
model.fit(train, validation_data=val, epochs=EPOCHS)

# Final report (precision / recall / F1 per class)
y_true, y_pred = [], []
for x, y in val:
    p = model.predict(x, verbose=0)
    y_true.extend(np.argmax(y.numpy(), axis=1))
    y_pred.extend(np.argmax(p, axis=1))

print(classification_report(y_true, y_pred, target_names=classes, digits=3))
print(confusion_matrix(y_true, y_pred))
model.save("urban_intel_cnn.keras")

12Troubleshooting

Every error you might hit during the demo, and exactly how to fix it.

Symptom Cause Fix
Register returns 503 "Database not initialized" Supabase tables don't exist yet. Run backend/supabase_setup.sql in the Supabase SQL editor.
Register returns 400 "already exists" That email was used before. Sign in instead, or delete the row from Supabase Table Editor → users.
/api/health shows "ai":"fallback" No AI key set. Add GEMINI_API_KEY from aistudio.google.com/apikey to backend/.env, restart.
Expo web shows blank screen EXPO_PUBLIC_BACKEND_URL wrong. Set frontend/.env.local to http://localhost:8001, then restart Expo.
Login from phone fails Phone can't reach localhost. Use your laptop's LAN IP: EXPO_PUBLIC_BACKEND_URL=http://192.168.x.x:8001.
AI triage takes > 5 s First call to Gemini = cold start. Click "Run AI Triage" once before the demo. Subsequent calls cache < 2 s.
Background GPS doesn't stream Expo Go can't run background tasks. This is by design — needs an eas build. In-app foreground streaming works fine for the demo.
